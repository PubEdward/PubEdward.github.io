深入内存对齐

深入内存对齐

  最近在一块旧芯片上实现交互协议，遇到了内存对齐问题，就此整理一下。

  内存对齐是指数据存储在内存时的起始地址是否是某个值的整数倍，本身是对软件开发透明的，编译器会把变量按本身的平台进行对齐。如果只是放在内存中，是否对其本身并没有什么问题，但是，在访问一个不对齐的数据(unaligned memory access)可能会导致效率问题，或崩溃。

  内存的对其和不对齐，是在时间和空间上的一个权衡。对齐节省了时间，假设一个体系结构的字长为w，那么它就假设了在这种体系结构上对宽度为w的数据的处理最频繁也是最重要的。它的设计也是从优先提高对w位数据操作的效率来考虑的。比如读写，大多数情况下需要读写w位数据，因而数据通道就会是w位。如果所有的数据访问都以w位对齐，那么访问就可以加快，因为需要传输的地址位减少，可以加快寻址。大多数体系结构都是按照字长来对齐访问数据的。不对齐的时候，有的会出错，比如MIPS上会产生bus error，而x86则会多次访问来拼接所得到的结果，从而降低执行效率。

  有些体系结构是必须对齐的，比如sparc、MIPS。它们在硬件的设计上就强制性的要求对齐。不是因为它们做不到不对齐，而是它们追求的是速度，这样没有意义。

  代码中关于对齐的问题，很多是隐式的，比如在强制类型转换的时候。在ARM架构下，旧的cpu在加载和存储数据时需要对齐的地址(有的cpu支持内存不对齐访问)，如下指令：

LDRB/STRB   address must be byte aligned
LDRH/STRH   address must be 2-byte aligned
LDR/STR     address must be 4-byte aligned

  有一块由char *p指向的缓冲区，在其中取出uint32_t的数据((uint32_t*)p)会调用LDR/STR指令，如果发生了内存不对齐的事件，较旧的cpu则不会正确运行，发生总线错误。安全的做法是采用memcpy，这时cpu会综合调用上述指令，不会关心对齐，数据按照字节流拷贝。

  还有一点需要提及，当用attribute(packed))修饰一个结构体时，编译器不会在其中插入任何补齐字节。可以用在表示离线的固实的数据的结构的修饰上。

  You might be inclined to believe that usage of this attribute can easily lead to unaligned accesses when accessing fields that do not satisfy architectural alignment requirements. However, again, the compiler is aware of the alignment constraints and will generate extra instructions to perform the memory access in a way that does not cause unaligned access. Of course, the extra instructions obviously cause a loss in performance compared to the non-packed case, so the packed attribute should only be used when avoiding structure padding is of importance.

  由此，常用的做法

#pragma pack(push,1)
struct NetPack
{
  //...
};
#pragma pack(pop)

是错误的

  目前的开发普遍重视性能，对齐的问题，有三种不同的处理方法：

  1. 采用对齐的方式声明。

  2. 对于逻辑上相关的成员变量希望放在靠近的位置，可以写成不对齐的方式，但需要显式的插入reserved成员。

  3. 交给编译器自动对齐
